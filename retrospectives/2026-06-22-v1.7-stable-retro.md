# v1.7-stable 复盘

> 日期：2026-06-22 | 从草案到 stable 的完整 loop

## v1.7 做对了什么

1. **Mode Router 方向正确**。"小任务过重"是一个真实断裂点——用户明确表达过、GAP 分析记录过。quick/full 分流直接解决了这个痛点。两个真实 quick 任务都在 ~1 分钟完成，vs full 模式 5 分钟。

2. **思考线 v4 审查有效**。原草案 5 模式 + 8 Gate + 6 目录重组，审查结论"砍半"——保留 2 模式 + 1 Gate + 单文件。如没有这次审查，v1.7 会变成一次扩张伪装的重构。

3. **"只加固断裂点"原则守住了**。三项验证后确认：quick 模式填补了真实断裂点；Conflict Gate 独立化是已有验证能力的结构化；7 个 placeholder Gate 没有进入强制流程——没断的墙没建。

4. **验证驱动的 tag 决策**。不是"文档写完了就打 tag"，而是跑完 3 个真实任务（1 回归 + 1 真实 quick + 1 full 功能）才打。每个任务有 evidence（commit + reports + Unity Play）。

## 原 v1.7 草案为什么过度

根本原因：把"架构清晰"等同于"功能更多"。

| 草案 | 实际需求 | 过度部分 |
|:---|:---|:---|
| 5 模式 | 只需要 2 种 | review/retro/research 是动作不是入口 |
| 8 Gate 全实现 | 只有 1 个断过 | 7 个在没断裂的点上建墙 |
| 6 目录重组 | 600 行单文件可管理 | 拆分协调税 > 收益 |
| 2 套 Lite 报告 | 只需要 1 套 | quick 模式只需要 WorklineSummary + metrics-lite |

**教训**：管线设计者（包括 AI Agent）有天然的"架构膨胀"倾向——把每个可能的问题都提前解决。Harness Engineering 的核心纪律就是对抗这种倾向。

## 思考线反方审查如何修正方向

三路攻击者各自命中要害：

- **管线原则攻击者**：5 项提案中仅 1.5 项对应真实断裂 → 砍到 2 模式 + 1 Gate
- **开发者攻击者**：TaskSpec 增加入口负担 → 改为 Agent 内部判断
- **架构师攻击者**：6 层抽象协调税 > 收益 → 保持单文件

**关键机制**：不是 Hermes 自己反思，而是让三个独立攻击者从不互看的立场攻击。攻击强度量化（强/中），被攻破的主张直接标记 ABANDONED。这个机制本身值得保留——以后每个大版本草案都应该跑一次。

## quick 模式真实收益

```
Step 1 (regression 06): ~1 min  — 单行恢复，Codex 15s 修复
Step 2 (real quick):    ~1 min  — 单行删除，Codex 23s 修复
Step 3 (full feature):  ~5 min  — 3 task 并行，SDD + Conflict + Review + 9 份报告
```

**quick 比 full 快 ~5x**。主要节省在：
- 无 SDD 撰写（省 2-3 min）
- 无 Conflict Gate 完整检查（省 1 min）
- 无 4 份结构化报告（省 2-3 min）
- 无 Claude Code 审查委托

## full 模式是否保持质量

Step 3 验证了 full 模式无退化：
- Conflict Gate 正常 PASS（7/7 检查通过）
- 3 个 Codex task 全部首次正确
- 审查 APPROVED，0 问题发现
- 用户验证发现 1 个 UI 问题（detailText 高度），修正后通过
- scope-creep: 0, violations: 0

质量未降。用户发现的 UI 问题是 SDD 阶段未检查 Text 行数容量——可沉淀为 SDD 模板检查项。

## 下一阶段是否真的需要 Unity UI Gate / Performance Gate

**不需要。** 三个证据：

1. Step 3 的唯一 bug（detailText 高度裁剪）不是 Gate 能拦截的——它是 SDD 模板缺少"UI Text 行数容量"检查项，不是缺少一个独立 Gate。
2. 7 个 placeholder Gate 在 3 个任务中没有一个被触发。它们等待的"断裂"尚未发生。
3. v1.7 的设计已经预留了扩展点——当真断裂发生时，复制 placeholder → 标记 active → 加入 regression。不需要提前建。

**v1.8 不应该做 Gate 扩展。应该让 v1.7 在真实使用中积累数据——至少再跑 3-5 个任务，看哪个 placeholder Gate 的触发条件首先满足。**
