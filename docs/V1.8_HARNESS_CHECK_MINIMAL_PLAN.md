* 自动检查器检查什么
* 什么情况 PASS / WARNING / BLOCKED
* 如何验证 v1.8 最小闭环

## 1. 一句话目标

v1.8-harness-check-minimal 的目标是：

> 将 v1.7-stable 的流程规则，从"文档约束"推进到"可执行检查"。

v1.8 要让系统自动回答：

1. 这个任务产物完整吗？
2. full / quick 模式是否合规？
3. metrics 是否完整？
4. 是否存在无关文件修改？
5. 是否跳过了必要验证？
6. result 是否与 build / review / user_verified 一致？
7. 这次任务能不能被标记为完成？

* full / quick 双模式

* Full / Lite 报告体系
* metrics.yaml / metrics-lite.yaml
* regression 06

* 真实 full 任务验证

## 3. 当前主要缺口

v1.7 的主要短板不是流程不够，而是：

```text
规则还主要靠文档和人记住。
```

具体缺口：

1. 报告是否齐全缺少自动检查；
2. metrics 字段是否完整缺少自动检查；
3. full / quick 是否合规缺少自动检查；
4. scope creep 主要靠人工看 diff；
5. REVIEW / TestReport / user_verified 是否缺失主要靠人工判断；
6. stable tag 前验收偏手动；
7. regression 尚未被统一纳入自动检查；
8. 文档规则还没有变成可执行规则。

结论：

> v1.8 的价值不是加更多流程，而是让已有流程具备执行力。

## 4. 为什么下一步不是完整真实工作流

当前已经有 `WORKLINE_GIT_WORKFLOW.md`，包含：

* main / develop
* feature / fix
* docs / skill / regression
* release / hotfix
* PR / MR
* review
* tag
* rollback

这已经足够让工作线适配真实公司 Git 流程。

下一步不优先做：

* 完整 CI/CD
* 完整 Release 系统
* 完整 QA 流程
* 多人协作模拟
* 线上监控
* 自动发版
* 复杂 Git 平台规则

原因：

1. 当前是个人系统；
2. 真实团队压力不存在；
3. Git Workflow 已经有适配层；
4. 继续模拟公司流程容易变成流程表演；
5. 当前更需要 Harness 检查能力。

结论：

```text
Git Workflow 已经够用。
下一步应该让 Harness 更硬。
```

## 7. 检查器输入

检查器需要支持以下输入：

```bash
python tools/check_workline_task.py --task tasks/<task-id> --mode full
python tools/check_workline_task.py --task tasks/<task-id> --mode quick
```

可选参数：

```bash
--base <commit-hash>
--changed-files <file>
--strict
--report tasks/<task-id>/HarnessCheckReport.md
```

输入信息：

* task_id
* mode
* task directory
* git base commit
* git diff
* changed files
* expected outputs
* metrics path
* regression status

## 8. full 任务检查规则

full 任务必须存在以下产物：

```text
TaskSpec.md
SDD.md
ConflictReport.md
REVIEW.md
ChangedFiles.md
TestReport.md
RiskReport.md
metrics.yaml
retrospective.md
```

### 8.1 文件完整性检查

缺失以下文件时直接 BLOCKED：

* SDD.md
* ConflictReport.md
* REVIEW.md
* TestReport.md
* metrics.yaml

缺失以下文件时 WARNING 或 BLOCKED，取决于 strict 模式：

* TaskSpec.md
* ChangedFiles.md
* RiskReport.md
* retrospective.md

---

### 8.2 REVIEW 检查

REVIEW.md 必须包含明确结论：

```text
APPROVED
NEEDS_FIX
REJECTED
```

规则：

* full 缺 REVIEW = BLOCKED
* REVIEW 无 verdict = BLOCKED
* REVIEW = REJECTED 但 result=approved = BLOCKED
* REVIEW = NEEDS_FIX 但无后续 fix 说明 = WARNING

---

### 8.3 TestReport 检查

TestReport.md 必须记录：

* build attempted
* build result
* test result
* Unity Play 是否验证
* known limitations

规则：

* build failed 但 result=approved = BLOCKED
* 未记录 build 状态 = BLOCKED
* Unity Play 未验证但文档宣称 completed = BLOCKED
* Unity Play pending 且 result=pending = PASS / WARNING

---

### 8.4 metrics.yaml 检查

metrics.yaml 必须包含：

```yaml
task_id:
mode:
result:
build:
  attempted:
  passed:
review:
  attempted:
  verdict:
scope:
  unrelated_files_changed:
  violations:
user_verified:
regression_required:
skill_patch_required:
```

规则：

* 缺关键字段 = BLOCKED
* mode 不是 full = BLOCKED
* result=approved 但 build.passed=false = BLOCKED
* result=approved 但 user_verified=false = BLOCKED
* scope.unrelated_files_changed > 0 = BLOCKED
* violations 非空但 result=approved = BLOCKED 或 WARNING，视 violation 等级决定

---

## 9. quick 任务检查规则

quick 任务必须存在：

```text
WorklineSummary.md
metrics-lite.yaml
```

### 9.1 文件完整性检查

规则：

* 缺 WorklineSummary.md = BLOCKED
* 缺 metrics-lite.yaml = BLOCKED

---

### 9.2 metrics-lite 检查

metrics-lite.yaml 必须包含：

```yaml
task_id:
mode:
result:
changed_files:
build:
  passed:
user_verified:
scope:
  violations:
auto_upgrade:
  required:
  reason:
```

规则：

* mode 不是 quick = BLOCKED
* result=approved 但 build.passed=false = BLOCKED
* result=approved 但 user_verified=false = BLOCKED
* scope.violations 非空但 result=approved = BLOCKED 或 WARNING
* auto_upgrade.required=true 但仍按 quick approved = BLOCKED

---

### 9.3 quick 越界检查

quick 不应涉及：

* 协议
* 配置表
* 资源
* 平台适配
* 大型 UI 结构
* 新依赖
* 多模块重构
* 公共 API 大改
* 大量文件修改

触发以下条件时应 auto-upgrade：

```text
changed_files > 3
涉及多个模块
涉及配置 / 协议 / 资源 / 平台
涉及新增依赖
涉及核心架构
涉及复杂 UI 结构
```

结果：

* 明确越界 = BLOCKED
* 可能越界 = WARNING
* 无越界 = PASS

---

## 10. Scope 检查规则

v1.8 先做路径级检查，不做复杂语义判断。

### 10.1 按任务类型检查路径

| 任务类型       | 允许路径                         | 禁止路径                                 |
| ---------- | ---------------------------- | ------------------------------------ |
| docs       | docs/ README.md              | skills/ regression/ source           |
| skill      | skills/                      | source unrelated docs                |
| regression | regression/                  | skills core source                   |
| full       | source + task reports        | unrelated docs / unrelated modules   |
| quick      | target source + lite reports | config / protocol / asset / platform |

---

### 10.2 scope 结果

```text
PASS
- changed files 在允许范围内

WARNING
- changed files 偏多，但仍可能合理
- 涉及边界文件，需要人工确认

BLOCKED
- docs 任务修改 source
- quick 修改协议 / 配置 / 资源 / 平台
- skill patch 修改业务代码
- regression 修改核心 skill
- unrelated_files_changed > 0
```

---

## 11. Regression 检查规则

v1.8 不实现大型 eval suite，但要检查 regression 状态是否被记录。

需要检查：

* regression/README.md 是否存在
* metrics 中是否有 regression_required
* metrics 中是否有 regression_status
* skill_patch_required=true 时是否说明 regression 是否需要新增
* stable tag 前是否要求 regression passed

允许状态：

```text
passed
failed
skipped
manual
not_required
```

规则：

* regression_required=true 但无 regression_status = BLOCKED
* regression_status=failed 但 result=approved = BLOCKED
* regression_status=skipped 且无原因 = WARNING
* regression_status=manual 且有说明 = PASS / WARNING

## 13. 阻断规则

### 13.1 BLOCKED

以下情况必须 BLOCKED：

* full 缺 SDD
* full 缺 REVIEW
* full 缺 TestReport
* full 缺 metrics.yaml
* quick 缺 WorklineSummary
* quick 缺 metrics-lite
* metrics 缺关键字段
* build failed 但 result=approved
* user_verified=false 但 result=approved
* REVIEW=REJECTED 但 result=approved
* unrelated_files_changed > 0
* quick 触发 auto-upgrade 但仍 approved
* regression failed 但 result=approved

---

### 13.2 WARNING

以下情况可 WARNING：

* retrospective 内容过短
* changed files 数量偏多但未越界
* regression skipped 但有原因
* TestReport 存在但验证说明不足
* RiskReport 存在但风险等级不清
* auto-upgrade 条件可疑但不确定

---

### 13.3 PASS

PASS 条件：

* 必要产物齐全
* metrics 字段完整
* result 与 build / review / user_verified 一致
* scope 无违规
* regression 状态合理
* 无 BLOCKED 项


`UNITY_CLIENT_LIFECYCLE_COVERAGE.md` 负责描述完整游戏客户端领域地图。

v1.8 不补完整游戏客户端生命周期。

```text

```

## 20. 风险与缓解


```text

```

---

### 风险 3：误判 quick / full

缓解：

```text

```

---

### 风险 4：Unity 验证无法自动化

```text
v1.8 只检查是否记录了 Unity 验证结果。
不强制自动启动 Unity。
```

---

### 风险 5：检查器和文档规则漂移

缓解：

```text
每次修改规则，必须更新检查器验证样例。
```
