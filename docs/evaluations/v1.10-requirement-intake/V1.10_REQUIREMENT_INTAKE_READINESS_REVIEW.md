## 0. Executive Summary

该文档定义了一个 **Requirement Intake Layer（需求入口层）**，用于在策划需求进入现有 workline 之前，先做结构化拆解和模糊点检测。

**结论：方向正确，设计克制，适合作为 v1.10。但建议先走最小实现路径（Option A），不做 runtime skill。**

它与 v1.7/v1.8/v1.9 的关系是**入口层**——不修改现有管线，不新增 Gate，不接 LangGraph/token/Dashboard。不会破坏现有工作线。

**建议进入设计精炼阶段，暂不进入完整实现。**

## 1. Understanding Check

v1.10 解决的核心问题是：

```
策划需求文档（模糊的自然语言）
→ 需求摘要（工程化表述）
→ 模糊点识别（AmbiguityReport，含 BLOCKER/HIGH/MEDIUM/LOW 四级）
→ 客户端影响面分析（14 维度：UI/数据/网络/资源/性能等）
→ 任务拆分（Epic→Feature→Task→Subtask）
→ 验收标准（可验证条件）
→ 模式路由（quick/full/split/clarification-only）
→ 移交包（ImplementationBrief）
→ 再进入现有 v1.7/v1.8/v1.9 管线执行
```

**它不等于**：策划文档进来自动完成全部功能。它只负责"翻译和拆解"，不负责"实现"。

|:---|:---|---:|:---|
| RequirementDigest.md | 需求摘要 | yes | 低 — 核心文档 |
| AmbiguityReport.md | 模糊点检测 | yes | 低 — 防返工关键 |
| ClientImpactMap.md | 客户端影响面 | yes | 中 — 14 维度可能过重 |

| RiskRegister.md | 风险登记 | optional | 低 — 可合并到 ImplementationBrief |

### 过重风险

8 份文档对于小需求（如"修改一个按钮文案"）过重。建议：

**最小版**（小需求）：RequirementDigest + AmbiguityReport + ModeRoutingDecision（3 份）
**完整版**（中大型需求）：全部 8 份

ImplementationBrief 和 RiskRegister 可以合并为一份。

## 7. Minimal Implementation Recommendation

### Option A: Documentation-only intake protocol

只写规则和模板，不写代码。Hermes 按模板手动生成 intake 产物。

**优点**：零代码、不破坏现有管线、即时可用。
**缺点**：手动执行，不能自动化。
**适合**：验证阶段。

### Option B: Template-based intake generator

新增一个小工具，根据输入需求生成固定产物。

**优点**：半自动化，减少手工。
**缺点**：需要维护代码、需要 CI 集成。
**适合**：验证通过后的 v1.10-stable。

### Option C: Full runtime skill

创建新的 skill（`skills/workline-intake/SKILL.md`）。

**优点**：完整集成。
**缺点**：当前阶段过重、增加 C 级证据风险。
**不适合**：当前阶段。

### 推荐

```
Option A — Documentation-only intake protocol

先用真实需求手工验证 1-2 次。
验证通过后再考虑 Option B。
Option C 暂不推荐。
```

## 10. Final Verdict

```
READY_FOR_DESIGN_REFINEMENT
```

### 原因

- 方向正确：解决真实问题（模糊需求→返工），不破坏现有管线
- 设计克制：明确 non-goals，不扩展架构，不新增 Gate
- 但尚未验证：所有设计停留在文档层（C 级证据），需要用真实需求验证后才能判定成熟度
- 建议先走 Option A（文档协议），验证 1-2 次后再考虑 Option B（工具化）

### 不建议立即进入完整实现

- 8 份产物需要真实验证来确认是否过重
- 14 维度的 ClientImpactMap 需要确认在实际使用中是否实用
- 模板和 skill 应等验证通过后再创建

### 下一步建议

1. 将 v1.10 plan 复制到 hermes-harness `docs/`
2. 找一个真实或模拟策划需求执行 intake
3. 验证 intake → workline → checker 全链路
4. 根据验证结果决定是否进入 Option B
