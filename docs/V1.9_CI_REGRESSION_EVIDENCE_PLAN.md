## 0. 一句话目标

v1.9 的目标是把 v1.8 的手动 checker 和有限样本，升级为**最小自动工程证据系统**。

```
手动证据 → 自动证据
少量样本 → 更多样本
Git 文档 → Git 执行记录
```

**明确**：

- v1.9 不扩展架构
- v1.9 不新增 Gate
- v1.9 不接 token 系统
- v1.9 只补 CI、regression、Git workflow 执行证据、真实任务证据

## 3. 最小 CI 设计

### 新增文件

```text
.github/workflows/workline-check.yml
```

### 触发条件

```yaml
on:
  push:
  pull_request:
  workflow_dispatch:
```

### CI Job 内容

```yaml
jobs:
  harness-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - name: Install dependencies
        run: pip install -r requirements-dev.txt
      - name: Run harness checker fixtures
        run: python tests/harness_check/run_harness_check_cases.py
      - name: Check real quick task
        run: python tools/check_workline_task.py --task tasks/v1.8-real-quick-dependency-message --mode quick
      - name: Check real full task
        run: python tools/check_workline_task.py --task tasks/v1.8-real-full-harness-case-runner --mode full
```

### 约束

- 不接复杂 matrix
- 不上传 artifact
- 不接 coverage
- 不接 deployment
- 不接 token/cost
- **第一版只证明 checker 可以自动运行**

## 4. Regression 扩大计划

当前 regression 6 个。v1.9 目标 10+。新增 4 个：

### regression 07：quick scope violation

**输入**：quick 任务修改无关文件（docs 任务修改 source）

**预期**：BLOCKED — scope.violations 非空

**阻断规则**：scope.violations 非空 + result=approved → BLOCKED

**对应风险**：quick 模式被滥用，scope creep

### regression 08：full metrics conflict — build failed

**输入**：result=approved 但 build.passed=false

**预期**：BLOCKED

**阻断规则**：result=approved + build.passed=false → BLOCKED

**对应风险**：编译失败被忽略

### regression 09：full missing multiple required reports

**输入**：full 任务缺 SDD + REVIEW + metrics.yaml

**预期**：BLOCKED（3 个 required 缺失）

**阻断规则**：缺 SDD/REVIEW/TestReport/metrics → BLOCKED

**对应风险**：报告体系被架空

### regression 10：quick auto-upgrade required but approved

**输入**：auto_upgrade.required=true 但 result=approved

**预期**：BLOCKED

**阻断规则**：auto_upgrade.required=true + result=approved → BLOCKED

**对应风险**：quick 越界未被拦截

### 新增后回归矩阵

| # | 名称 | 模式 | 预期 | 状态 |
|:--|:---|:--:|:---|:--:|
| 01-05 | existing 5 cases | — | — | ✅/⚠️ |
| 06 | quick bugfix | quick | PASS | ✅ |
| 07 | quick scope violation | quick | BLOCKED | 待创建 |
| 08 | full metrics conflict | full | BLOCKED | 待创建 |
| 09 | full missing reports | full | BLOCKED | 待创建 |
| 10 | quick auto-upgrade bypass | quick | BLOCKED | 待创建 |

## 5. Git Workflow 执行演练

### 目标

当前 Git Workflow 文档已有完整分支模型，但从未真实执行。v1.9 至少执行一次轻量演练。

### 推荐任务

```text
fix/v1.9-ci-readme-typo
```

或

```text
feature/v1.9-regression-runner
```

### 演练流程

```text
develop
→ feature/<task-id> 或 fix/<task-id>
→ 运行 checker / regression
→ 生成 WorklineSummary 或 full reports
→ merge 回 develop
```

### 证据记录

如果没有远程 PR 系统，使用 PR-like markdown 记录：

```text
tasks/<task-id>/PullRequest.md
```

必须记录：

- source branch
- target branch
- changed files
- checks passed
- review checklist
- merge method

## 8. 风险

| 风险 | 说明 | 缓解 |
|:---|:---|:---|
| GitHub Actions 无法本地触发 | 本地无 GitHub runner | 先用 act 或直接 push 测试 |
| CI 配置变成形式主义 | 只跑一次就不再关注 | CI 结果写入 evidence report |
| regression case 增加但质量不高 | 为凑数而加 case | 每个 case 必须对应真实风险 |
| Git workflow 演练偏模拟 | 仍然是单人操作 | 记录 PR-like markdown 作为证据 |
| v1.9 范围滑向扩展 | 被诱导加入 token/Gate/Dashboard | 严格对照本文 scope |
| CI 引入依赖环境问题 | Python/PyYAML 版本差异 | 使用 actions/setup-python 固定版本 |
| checker 对历史任务路径依赖 | 移动 tasks/ 后 CI 失败 | 使用相对路径 + CI-only fixtures |

## 9. 最终结论

v1.9 不解决"系统更聪明"，只解决"系统更可验证"。

v1.9 的成功标志不是更多功能，而是：

```
手动证据 → 自动证据
手动 checker → CI 自动运行
少量样本（6）→ 更多样本（10+）
Git 文档 → Git 执行记录
```

v1.9 完成后，P4/P5/P6 分数应显著提升，A/B 级证据占比从 63% 提升至 ~80%，C/D/E 级从 36% 降至 ~20%。届时再提交思考线审判，判断是否达到 Stable Personal Harness。
